## Summary

This research covers four incremental features for Receptra v1.1, all built atop the Phase 4 RAG
foundation. The codebase was read in full before writing anything here. The architecture is tight
and well-designed; each feature has a clear insertion point with low blast radius.

**Primary recommendation:** Each feature is a narrow surgical extension:
Feature 1 lives entirely in `retriever.py` + `config.py` (already 90% done — `rag_min_similarity`
exists at 0.35; the feature is raising the default and piping the score through to the frontend).
Feature 2 is a new `extract_text()` function in `ingest.py` + two library deps.
Feature 3 is a new HTTP endpoint + a new prompt function in `prompts.py`.
Feature 4 is a new WS event type + a parallel `asyncio.create_task` in `hot_path.py`.

## Feature 1: Confidence-Gated Suggestions

### Current State (VERIFIED: codebase read)

The retriever already filters on `rag_min_similarity` (default 0.35) — `retriever.py` line 76:

```python
similarity = 1.0 - float(dist)  # cosine: distance → similarity
if similarity < threshold:
    continue
```

`settings.rag_min_similarity = 0.35` is defined in `config.py`. The retriever already returns an
empty list when nothing passes, which causes the engine to emit the canonical `אין לי מספיק מידע`
refusal. The missing piece is:

1. A **higher default threshold** (0.35 is too permissive for "confidence-gated" UI behaviour).
2. **Passing similarity scores to the frontend** so it can render a "low confidence" badge even
   when chunks were returned but scored low.
3. An env-var-configurable **suggestion-level threshold** (separate from the retrieval threshold).

### Distance Math (VERIFIED: codebase + ChromaDB docs)

The collection is configured with `metadata={"hnsw:space": "cosine"}` (vector_store.py line 81).
BGE-M3 emits L2-normalized vectors (confirmed by model card [ASSUMED]).

ChromaDB cosine distance = `1 - cosine_similarity`. So the existing conversion is correct:

```python
similarity = 1.0 - float(dist)  # already in retriever.py
```

- Distance 0.0 → similarity 1.0 (identical vectors)
- Distance 0.35 → similarity 0.65
- Distance 0.65 → similarity 0.35 (current floor — very permissive)

### Two-Level Threshold Design

There are two semantically distinct thresholds:

| Name | Current setting | Purpose |
|------|-----------------|---------|
| `rag_min_similarity` (0.35) | retrieval floor | Don't even return chunks this irrelevant |
| `rag_suggestion_threshold` (new, 0.65 default) | UI confidence gate | Show "low confidence" badge or suppress |

These are **different knobs** because:
- Chunks below 0.35 are noise — don't show them to the LLM.
- Chunks 0.35–0.65 may still produce useful responses but warrant a badge.
- Chunks above 0.65 are shown without a badge.

The retrieval floor stays at 0.35. The UI gate (0.65) is the new feature.

### Threshold Value Guidance (MEDIUM confidence)

For multilingual models like BGE-M3 applied to Hebrew, community consensus:
- Below 0.3: noise, filter at retrieval level (current floor)
- 0.3–0.5: semantically related but not directly answering
- 0.5–0.7: good matches, reasonable confidence
- Above 0.7: strong matches

0.65 as the suggestion-gating default is consistent with standard RAG practice for production
systems. The existing `rag_min_similarity=0.35` stays as the retrieval-level floor.
[CITED: FlagEmbedding repo — BGE threshold guidance; standard RAG practice per multiple sources]

### Where to Insert (VERIFIED: codebase)

**Backend changes:**

1. **`config.py`** — add one new field:
```python
rag_suggestion_threshold: float = 0.65
# Suggestion-confidence gate: if the max chunk similarity below this,
# the frontend renders a "low confidence" badge. Set to 0.0 to disable gating.
# Distinct from rag_min_similarity (retrieval-level noise filter).
```

2. **`retriever.py`** — `retrieve()` already returns `ChunkRef` objects. Add `max_similarity`
to the return value OR surface it in `source` dict (already has `"similarity": f"{similarity:.3f}"`
on line 88). No ChunkRef schema change needed — `source["similarity"]` carries it through.

3. **`hot_path.py`** — after `retrieve()` call, compute `max_similarity` from the returned chunks:
```python
max_similarity = max(
    (float(c.source["similarity"]) for c in chunks if c.source), default=0.0
)
low_confidence = bool(chunks) and max_similarity < settings.rag_suggestion_threshold
```
Pass `low_confidence` through to the `SuggestionComplete` event (requires schema extension).

4. **`pipeline/events.py`** — extend `SuggestionComplete`:
```python
rag_max_similarity: float = Field(default=0.0, description="Max chunk cosine similarity; 0.0 if no chunks")
rag_low_confidence: bool = Field(default=False, description="True if max_similarity < rag_suggestion_threshold")
```

5. **`frontend/src/types/ws.ts`** — mirror the two new fields in `SuggestionComplete`.

6. **`SuggestionPanel.tsx`** — render a badge:
```tsx
{suggestions.rag_low_confidence && (
  <div className="rounded-md border border-yellow-200 bg-yellow-50 px-2 py-1 text-xs text-yellow-700">
    ⚠ ביטחון נמוך — ייתכן שהמידע חלקי
  </div>
)}
```

**API exposure (optional for v1.1):** The threshold is already env-var configurable via
`RECEPTRA_RAG_SUGGESTION_THRESHOLD`. No REST endpoint needed for v1.1.

### Config Schema


iteration works correctly for Hebrew text content even if RTL rendering metadata is lost.
[VERIFIED: pip index versions]

### Hebrew PDF RTL Extraction Gotchas (MEDIUM confidence)

Hebrew PDFs created by digital tools (Word exports, Acrobat) encode text in one of two ways:

1. **Logical order** (correct): text stored left-to-right in internal order, renderer handles RTL.
   `pypdf` extracts correctly; no post-processing needed.
2. **Visual order** (problematic): text stored physically right-to-left as rendered on page.
   Words come out in correct visual order but characters within each word may be reversed.

There is **no reliable flag in `pypdf` to detect which encoding was used**. [ASSUMED]

The standard defense is to run `python-bidi` 0.6.7 `get_display()` on each extracted line after
page concatenation, which re-applies the Unicode BiDi algorithm. This handles both cases:
- Logical-order PDFs: no change (bidi algorithm is idempotent for LTR-stored Hebrew).
- Visual-order PDFs: reverses scrambled character order back to reading order.

```python
from bidi import get_display

def fix_rtl(text: str) -> str:
    """Apply Unicode BiDi algorithm to fix visual-order Hebrew PDF extraction."""
    lines = text.splitlines()
    return "\n".join(get_display(line) for line in lines if line.strip())
```

`python-bidi` 0.6.7 uses a Rust backend by default (fast) with a pure Python fallback.
License: LGPL-2.1 [VERIFIED: pip index]. LGPL is **acceptable** for use as a library dependency
when not modifying the library itself.

**Installation:**
```bash
uv add "pypdf>=6.10.2,<7" "python-docx>=1.2.0,<2" "python-bidi>=0.6.7,<1"
```

### Extraction Patterns

**PDF extraction with `pypdf`:**
```python
# Source: pypdf docs + verified against pypdf 6.x API
import pypdf
from bidi import get_display

def extract_pdf_text(content: bytes) -> str:
    reader = pypdf.PdfReader(io.BytesIO(content))
    pages: list[str] = []
    for page in reader.pages:
        raw = page.extract_text(extraction_mode="layout")
        if raw:
            fixed = get_display(raw)
            pages.append(fixed)
    return "\n\n".join(pages)
```

Note: `extraction_mode="layout"` (added in pypdf 4.0) preserves spatial layout, which helps with
tabular data like price lists. [VERIFIED: pypdf 6.x changelog]

**DOCX extraction with `python-docx`:**
```python
import docx  # python-docx import name

def extract_docx_text(content: bytes) -> str:
    doc = docx.Document(io.BytesIO(content))
    paragraphs = [p.text for p in doc.paragraphs if p.text.strip()]
    return "\n\n".join(paragraphs)
```

Hebrew DOCX paragraph text is in logical (correct) order. `python-docx` paragraph iteration
is reliable; RTL formatting metadata (alignment, direction) is ignored — we only need the
text content for embedding. [VERIFIED: python-docx 1.2.0 API — doc.paragraphs is documented]

### Insertion Point in `ingest.py` (VERIFIED: codebase)

Current `ingest.py` has:
- `ALLOWED_EXTS: frozenset[str] = frozenset({".md", ".txt"})` (line 30)
- `_validate_extension()` checks against this set (line 34)
- `_decode_utf8_strict()` decodes to str (line 52)
- `ingest_document()` calls `_validate_extension → _validate_size → _decode_utf8_strict → chunk_hebrew`

**Change:** Add a dispatch function that replaces `_decode_utf8_strict` for binary formats:

```python
ALLOWED_EXTS: frozenset[str] = frozenset({".md", ".txt", ".pdf", ".docx"})

def _extract_text(filename: str, content: bytes) -> str:
    """Extract plain text from content bytes based on file extension."""
    lower = filename.lower()
    if lower.endswith(".pdf"):
        return _extract_pdf(content)
    elif lower.endswith(".docx"):
        return _extract_docx(content)
    else:
        return _decode_utf8_strict(content)

def _extract_pdf(content: bytes) -> str:
    import io
    import pypdf
    from bidi import get_display
    try:
        reader = pypdf.PdfReader(io.BytesIO(content))
    except Exception as e:
        raise IngestRejected(code="encoding_error", detail=f"PDF parse failed: {e}") from e
    pages: list[str] = []
    for page in reader.pages:
        raw = page.extract_text(extraction_mode="layout") or ""
        if raw.strip():
            pages.append(get_display(raw))
    text = "\n\n".join(pages)
    if not text.strip():
        raise IngestRejected(code="encoding_error", detail="PDF yielded no extractable text (scanned PDF?)")
    return text

def _extract_docx(content: bytes) -> str:
    import io
    import docx
    try:
        doc = docx.Document(io.BytesIO(content))
    except Exception as e:
        raise IngestRejected(code="encoding_error", detail=f"DOCX parse failed: {e}") from e
    paragraphs = [p.text for p in doc.paragraphs if p.text.strip()]
    if not paragraphs:
        raise IngestRejected(code="encoding_error", detail="DOCX yielded no paragraph text")
    return "\n\n".join(paragraphs)
```

In `ingest_document()`, replace line `text = _decode_utf8_strict(content)` with
`text = _extract_text(filename, content)`.

### Schema Changes

`IngestRejected` error codes already include `"encoding_error"` — no new error code needed.
`KbErrorResponse` Literal already includes `"encoding_error"`.

**`schema.py`:** Update the `KbErrorResponse.code` Literal to add `"unsupported_format"` if
scanned-PDF-no-text is a distinct UX concern; otherwise reuse `"encoding_error"`.

### Chunking Strategy for Longer Documents

Existing `chunk_hebrew()` handles arbitrarily long documents — no change needed. The 1 MiB
`MAX_BYTES` limit stays. PDFs of legal docs / price lists typically fit well under 1 MiB as
bytes. If a PDF is larger, the `_validate_size` check raises `file_too_large`. [VERIFIED: ingest.py]

### API Contract Preservation

The `/api/kb/upload` endpoint (multipart) and `/api/kb/ingest-text` endpoint (JSON text) are
unchanged in signature. The upload endpoint already dispatches to `ingest_document()` which
accepts a filename + bytes. Adding `.pdf`/`.docx` to `ALLOWED_EXTS` is backward-compatible:
`.md`/`.txt` still work identically.

## Feature 3: Post-Call Summary

### Trigger Architecture (VERIFIED: codebase)

The `finals: FinalTranscript[]` array in `useWebSocket.ts` accumulates all completed utterances
for the session. The full transcript is available on the frontend.

**Design decision:** POST to a new HTTP endpoint — do NOT use the WS connection for this.

Rationale:
- Summary is a one-shot, not a stream of events.
- The WS connection may be closed when the agent clicks "סיים שיחה".
- A separate HTTP call allows streaming the summary response back as Server-Sent Events or as
  a single JSON response (simpler for v1.1: non-streaming, return full text).

**Endpoint:** `POST /api/summary` — outside the `/api/kb` router, registered directly on `app`.

### Token Budget Calculation (VERIFIED: config.py)

```
llm_num_ctx: int = 8192  # from config.py
```

Voice conversations generate ~125–150 tokens per minute [CITED: AssemblyAI blog]. A 10-minute
call ≈ 1250–1500 tokens of transcript. Even with system prompt + few-shot overhead (~500 tokens),
an 8192-context window handles calls up to ~45 minutes. For v1.1, no truncation logic is needed.

However, the existing `MAX_TRANSCRIPT_CHARS = 2000` in `prompts.py` is for per-utterance
transcripts. The summary prompt gets the **full call transcript**, which will be much larger.
The summary prompt must NOT use `build_messages()` — it needs its own message builder with
a larger character budget:

```
Max transcript for summary: ~12000 chars (≈ 3000 Hebrew tokens) — leaves ~5000 tokens for
response + system prompt overhead within the 8192 token context.
```

If a transcript exceeds 12000 chars, truncate to the most recent 12000 chars (end of call
is most relevant for action items).

### Endpoint Pattern (VERIFIED: codebase engine pattern)

```python
@router.post("/summary", response_model=CallSummary)
async def generate_call_summary(body: CallSummaryRequest) -> CallSummary:
    full_transcript = "\n".join(body.transcript_lines)
    if len(full_transcript) > 12000:
        full_transcript = full_transcript[-12000:]  # tail truncation

    messages = [
        {"role": "system", "content": SYSTEM_PROMPT_SUMMARY_HE},
        {"role": "user", "content": _SUMMARY_FEW_SHOT_USER},
        {"role": "assistant", "content": _SUMMARY_FEW_SHOT_ASST},
        {"role": "user", "content": f"<transcript>\n{full_transcript}\n</transcript>"},
    ]
    # Call Ollama non-streaming (simpler for one-shot summary)
    client = get_async_client()
    chosen_model = await select_model(client)
    t0 = time.perf_counter()
    response = await client.chat(
        model=chosen_model,
        messages=messages,
        stream=False,
        options={"temperature": 0.0, "num_predict": 1024, "num_ctx": 8192},
    )
    total_ms = int((time.perf_counter() - t0) * 1000)
    raw = response.message.content or ""
    cleaned = _strip_markdown_fences(raw)
    parsed = CallSummary.model_validate_json(cleaned)  # raises on bad JSON
    return parsed.model_copy(update={"raw_text": raw, "model": chosen_model, "total_ms": total_ms})
```

Use `_strip_markdown_fences` from `receptra.llm.engine` (already exported in `__all__`).

## Feature 4: Real-Time Intent Detection

### Approach: Few-Shot Prompt to DictaLM (RECOMMENDED)

Two options:

| Approach | Latency | Accuracy | Implementation effort |
|----------|---------|----------|----------------------|
| Few-shot prompt to DictaLM (same Ollama) | +200–500ms | Good for 6 categories | Low — reuse existing client |
| Dedicated classifier (HebELECTRA, etc.) | +50–100ms | Potentially better | High — new model, new runtime |

**Recommendation: few-shot to DictaLM.** The latency cost is acceptable because intent detection
runs in **parallel** with the suggestion pipeline (see below). A dedicated classifier would require
managing a second model download, dependency on transformers or onnxruntime, and more memory pressure
on the 16GB unified memory budget. [ASSUMED — latency estimate; verify on reference M2]

### Parallelism Pattern (VERIFIED: codebase stt/pipeline.py)

Current flow (sequential):
```
FinalTranscript → await suggest(transcript, ...) → done
```

New flow (parallel):
```
FinalTranscript → asyncio.gather(suggest(...), detect_intent(...)) → done
```

Both results arrive before the loop continues. The intent badge appears on the frontend almost
simultaneously with the suggestion stream.

```python
# In hot_path.py or pipeline.py, replace:
await suggest(text, t_speech_end_ms, utterance_id)

# With:
await asyncio.gather(
    suggest(text, t_speech_end_ms, utterance_id),
    detect_intent_and_send(text, ws),
    return_exceptions=True,  # intent failure must not crash suggest pipeline
)
```

`detect_intent_and_send` is a new coroutine that calls Ollama with a fast intent prompt and sends
one `intent_detected` WS event.

```python
    "מידע": "information",
    "ביטול": "cancellation",
    "אחר": "other",
}

async def detect_intent(transcript: str) -> str:
    """Return one of: booking, complaint, billing, information, cancellation, other."""
    messages = [
        {"role": "system", "content": SYSTEM_PROMPT_INTENT_HE},
        {"role": "user", "content": _INTENT_FEW_SHOT_1_USER},
        {"role": "assistant", "content": _INTENT_FEW_SHOT_1_ASST},
        {"role": "user", "content": _INTENT_FEW_SHOT_2_USER},
        {"role": "assistant", "content": _INTENT_FEW_SHOT_2_ASST},
        {"role": "user", "content": _INTENT_FEW_SHOT_3_USER},
        {"role": "assistant", "content": _INTENT_FEW_SHOT_3_ASST},
        {"role": "user", "content": transcript[:500]},  # DoS guard
    ]
    client = get_async_client()
    chosen_model = await select_model(client)
    response = await client.chat(
        model=chosen_model,
        messages=messages,
        stream=False,
        options={"temperature": 0.0, "num_predict": 10},  # 1 word only
    )
    raw = (response.message.content or "אחר").strip()
    return INTENT_LABELS.get(raw, "other")
```

`num_predict=10` limits the response to essentially one word — fast and cheap.

### Backward Compatibility (VERIFIED: codebase)

The `useWebSocket.ts` `switch (event.type)` pattern has a `default` path (the switch falls through
to no-op for unrecognized types via TypeScript's exhaustive check on the union). Adding a new
discriminant to the union is backward-compatible: old clients simply have no `case 'intent_detected'`
and the event is ignored. Adding it to the union and handling it in the switch is non-breaking for
any future downgrade scenario.

### Intent Badge in TranscriptPanel

Each `FinalTranscript` in the `finals` array corresponds to one utterance. The intent badge should
appear on the corresponding utterance card. The simplest approach: maintain a separate
`intents: Record<number, IntentDetected>` state keyed by utterance index, updated when
`intent_detected` arrives.

```tsx
// In useWebSocket.ts:
const [intents, setIntents] = useState<Record<string, IntentDetected>>({})

// In onmessage:
case 'intent_detected':
  setIntents((prev) => ({ ...prev, [event.utterance_id]: event }))
  break
```

Then in `TranscriptPanel.tsx`, match each final to its intent by `utterance_id` — requires
`FinalTranscript` to carry `utterance_id`. Currently it does not include `utterance_id`.

**Simpler alternative for v1.1:** Show the **most recent intent** as a global badge on the panel
header rather than per-utterance. This avoids modifying the `FinalTranscript` schema:

```tsx
{currentIntent && (
  <span className="rounded bg-blue-100 px-2 py-0.5 text-xs text-blue-800">
    {currentIntent.label_he}
  </span>
)}
```

## Common Pitfalls

### Pitfall 1: PyMuPDF AGPL License Violation
**What goes wrong:** Developer installs `pymupdf` / `fitz` — commonly recommended for Hebrew PDFs —
without checking license. AGPL-3.0 requires open-sourcing the application.
**Prevention:** Use `pypdf` (BSD-3-Clause). Enforce in CI: `pip-licenses --fail-on="AGPL"`.

### Pitfall 2: Visual-Order Hebrew PDFs Without Bidi Fix
**What goes wrong:** `pypdf` extracts Hebrew characters in reversed order on some PDFs (those
encoded in visual order). Text reads right-to-left on screen but characters within each word
are stored in that visual (reversed-physical) order.
**Prevention:** Always run `bidi.get_display()` on extracted lines. It's idempotent on correct text.

### Pitfall 3: Summary Endpoint Bypasses Existing `build_messages` DoS Guards
**What goes wrong:** The existing `MAX_TRANSCRIPT_CHARS = 2000` guard in `prompts.py` is for
per-utterance transcripts. The summary receives the FULL call transcript which could be 10,000+
chars. Using `build_messages()` would raise `ValueError` for any call > 2000 chars.
**Prevention:** The summary endpoint has its OWN message builder with a separate character limit
(12000 chars recommended). NEVER call `build_messages()` from the summary endpoint.

### Pitfall 4: Intent Detection Serialized Behind Suggestion Pipeline
**What goes wrong:** If `detect_intent` is called sequentially after `suggest`, the agent sees the
intent badge only after the full suggestion pipeline completes (~1.5–2s). The badge becomes
useless at that point.
**Prevention:** Use `asyncio.gather(suggest(...), detect_intent_and_send(...))`. Both coroutines
run concurrently on the same event loop thread. Ollama handles both chat calls; the 12B model
can process two concurrent requests [ASSUMED — verify with Ollama's parallelism on M2].

### Pitfall 5: `asyncio.gather` Swallows Intent Failures Silently
**What goes wrong:** With `return_exceptions=True`, a raised exception in `detect_intent_and_send`
is returned as a result object, not re-raised. If not checked, failures vanish.
**Prevention:** After `gather`, check results for exceptions and log them:
```python
results = await asyncio.gather(suggest(...), detect_intent_and_send(...), return_exceptions=True)
for r in results:
    if isinstance(r, Exception):
        logger.bind(event="pipeline.intent_error").warning({"err": str(r)})
```

### Pitfall 6: DictaLM May Not Strictly Return One Hebrew Word for Intent
**What goes wrong:** The intent prompt asks for one word. DictaLM may add punctuation, a period,
or emit "הזמנה." (with period) or "הזמנת" (with suffix). The lookup `INTENT_LABELS.get(raw, "other")`
then falls to "other" unexpectedly.
**Prevention:** Normalize the raw response before lookup:
```python
raw = (response.message.content or "אחר").strip().rstrip(".,!?:")
# Also try prefix match:
for he_label, en_label in INTENT_LABELS.items():
    if raw.startswith(he_label):
        return en_label
return "other"
```

### Pitfall 7: DOCX Tables Not Extracted by `doc.paragraphs`
**What goes wrong:** Price list DOCX files often use tables. `doc.paragraphs` iterates only
paragraph-level text; table cell content is skipped entirely.
**Prevention:** Also iterate `doc.tables`:
```python
paragraphs = [p.text for p in doc.paragraphs if p.text.strip()]
for table in doc.tables:
    for row in table.rows:
        for cell in row.cells:
            if cell.text.strip():
                paragraphs.append(cell.text.strip())
return "\n\n".join(paragraphs)
```

## Open Questions

1. **Ollama parallelism on M2:** Can DictaLM 3.0 handle two concurrent chat calls (suggest +
   intent) on a 16GB M2? If not, intent detection may need to be fire-and-forget post-suggest.

2. **Scanned PDFs:** The target documents (clinic price lists, legal docs) — are any scanned?
   If yes, OCR (via `tesseract` + `pytesseract` or `easyocr`) would be needed. This is a
   significant scope increase; should be deferred unless users report empty extractions.

3. **Per-utterance intent vs. global badge:** v1.1 recommendation is a global "most recent intent"
   badge. If per-utterance is required, `FinalTranscript` must be extended with `utterance_id`
   (currently not on the wire event). This is a WS schema change with broader impact.

4. **Summary streaming:** The non-streaming `POST /api/summary` is simpler. If the summary model
   is slow on longer calls, users may see a loading spinner for 3–5s. Streaming via SSE or
   returning the summary as a WS event is possible but adds complexity. Defer to user feedback.
