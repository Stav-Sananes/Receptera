    Create two files.

    **File 1: `.github/workflows/license-gate-test.yml`** — exact content.

    Design: This workflow is `workflow_dispatch` only (manual trigger, per OPEN-8). It installs a known-GPL PyPI package (`gnureadline`, which declares `GNU General Public License v3 (GPLv3)`) into a scratch venv, runs the allowlist with the same string, and asserts the check exits non-zero. We do NOT run this on every commit because (a) it slows CI, (b) it pollutes the cache with a known-bad package, and (c) its job is to catch regressions in the gate configuration itself — a weekly or pre-release cadence is sufficient.

    ```yaml
    # Receptra — license gate regression test.
    #
    # Manually-triggered workflow that proves scripts/check_licenses.sh actually
    # rejects a package with a disallowed license. Without this, "we have a
    # license check" is theater — the gate could be silently broken by a config
    # drift and we'd never know.
    #
    # Per research OPEN-8: manual workflow_dispatch, not on every commit.
    # Cadence: run before releases or when the license allowlist changes.

    name: License Gate Regression Test

    on:
      workflow_dispatch:

    jobs:
      gate-rejects-gpl:
        name: Gate rejects GPL package
        runs-on: ubuntu-latest
        steps:
          - uses: actions/checkout@v4

          - name: Install uv
            uses: astral-sh/setup-uv@v4

          - name: Pin Python version
            run: uv python install 3.12

          - name: Create scratch venv
            run: uv venv /tmp/scratch-venv

          - name: Install known-GPL package (gnureadline — GPLv3)
            run: /tmp/scratch-venv/bin/python -m pip install gnureadline pip-licenses

          - name: Run allowlist — MUST fail because gnureadline is GPLv3
            id: gate
            continue-on-error: true
            run: |
              /tmp/scratch-venv/bin/pip-licenses --allow-only="Apache Software License;Apache 2.0;Apache-2.0;MIT License;MIT;BSD License;BSD-3-Clause;BSD-2-Clause;ISC License;ISC;Python Software Foundation License;PSF-2.0;The Unlicense;Mozilla Public License 2.0 (MPL 2.0)"

          - name: Assert gate exited non-zero
            run: |
              if [ "${{ steps.gate.outcome }}" = "failure" ]; then
                echo "✓ License gate correctly REJECTED the GPL package."
                exit 0
              else
                echo "✗ License gate INCORRECTLY ACCEPTED a GPL package. The allowlist is broken." >&2
                exit 1
              fi

          - name: Inverse sanity check — empty scratch allows MIT
            run: |
              uv venv /tmp/sanity-venv
              /tmp/sanity-venv/bin/python -m pip install requests pip-licenses
              /tmp/sanity-venv/bin/pip-licenses --allow-only="Apache Software License;Apache 2.0;Apache-2.0;MIT License;MIT;BSD License;BSD-3-Clause;BSD-2-Clause;ISC License;ISC;Python Software Foundation License;PSF-2.0;The Unlicense;Mozilla Public License 2.0 (MPL 2.0);Charset-Normalizer"
              echo "✓ License gate correctly ACCEPTED permissive packages (requests + deps)."
    ```

    Notes on the inverse sanity check: `requests` pulls in `charset-normalizer` (MIT) and `urllib3` (MIT) and `idna` (BSD) — all permissive. Including the long form "Charset-Normalizer" in the allowlist string guards against a reporter quirk if HF changes how charset-normalizer's license is reported.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer ↔ repo | Developers push code; CI is the gate between a commit and a mergeable state. |
| CI runner ↔ PyPI/npm | CI installs deps from public registries. Malicious deps can slip in — license allowlist is the first-line filter; dependency pinning (uv.lock, package-lock.json) is the second. |
| CI runner ↔ repo secrets | Phase 1 CI uses NO repo secrets. Nothing in ci.yml reads `secrets.*`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06-01 | Supply Chain | Disallowed-license dep merged without CI noticing | mitigate | Three layers of defense: (a) `scripts/check_licenses.sh` runs on every push (automatic block), (b) `license-gate-test.yml` proves the gate is functional (manual regression), (c) `uv.lock` + `package-lock.json` pin versions so adversarial dep upgrades can't sneak in. |
| T-01-06-02 | Tampering | CI workflow itself modified by a malicious PR to weaken gates | mitigate | GitHub Actions evaluates workflow YAML from the target branch, not the PR branch, for `pull_request` events — meaning a PR cannot modify its own CI behavior. GitHub enforces this at the platform level; no extra config needed. |
| T-01-06-03 | Denial of Service | CI concurrency storm from rapid pushes | mitigate | `concurrency.cancel-in-progress: true` cancels superseded runs, capping spend. |
| T-01-06-04 | Tampering | OPEN-1 regression (someone adds an ollama service to docker-compose.yml) | mitigate | The `compose` job has an explicit regression guard: `grep -qE "^\s*ollama:" docker-compose.yml` fails the build. |
| T-01-06-05 | Information Disclosure | pip-licenses / license-checker output logged publicly includes full dep tree | accept | Dep tree is implicitly public in a source-published OSS project; no incremental disclosure. |
| T-01-06-06 | Tampering | License gate silently broken by a typo in the allowlist (e.g., `MIT` → `MIT License` missing) | mitigate | `license-gate-test.yml` is the canary. Run before every release or after any change to `scripts/check_licenses.sh`. Without this the gate could degrade unnoticed. |
| T-01-06-07 | Elevation of Privilege | Third-party actions (setup-uv, setup-node, checkout) could be compromised | mitigate | Actions pinned to major versions (`@v4`). For higher assurance in later phases, pin to commit SHAs. Accepted tradeoff for Phase 1 given these are maintainer-trusted actions. |
</threat_model>
